# Agent 5 패턴 — 개념 + 프로젝트 매핑

Anthropic "Building effective agents" (2024.12) 의 5가지 패턴 정리.
**핵심 메시지**: agent 부터 의심하지 말고 chaining 부터 의심하라. 위로 갈수록 단순·통제 가능, 아래로 갈수록 자율·예측 불가.

---

## 1. Prompt Chaining (직선 파이프라인)

```
[Input] → LLM A → 중간결과 → LLM B → 중간결과 → LLM C → [Output]
```

Task 를 **고정된 단계 순서** 로 쪼개고 LLM 호출을 직선으로 연결.

**언제**: 단계가 명확히 정해져 있고 순서 안 바뀔 때. 단계 사이에 검증 (gate) 끼우기 좋음.

**예시 프로젝트**:
- **OCR 필드 추출 파이프라인** — 영수증 이미지 → OCR → 필드 추출 → 검증 → DB 저장
- **회의록 처리** — 음성 → STT → 요약 → 액션아이템 추출 → Slack 포맷
- **블로그 자동화** — 주제 → 개요 → 본문 → SEO 메타데이터
- **번역 파이프라인** — 번역 → 자기검수 → 용어통일

**함정**: 단계마다 LLM 호출 → latency 합산. 합칠 수 있는 단계는 합쳐.

## 2. Routing (분기 라우팅)

```
                    ┌→ 간단 task → 작은모델 (Haiku)
[Input] → 분류기 LLM ─┼→ 코드 task → 코드특화모델
                    └→ 복잡 task → 큰모델 (Opus)
```

입력을 분류해서 **각자 잘하는 핸들러로 보내는** 패턴. 비용·품질 둘 다 최적화 가능.

**언제**: 입력 종류가 다양해서 한 prompt/모델로는 비효율적일 때.

**예시 프로젝트**:
- **고객지원 봇** — 분류기 → FAQ 봇 (Haiku) / 환불 처리 (Sonnet) / 기술이슈 (Opus + 도구) / 사람 에스컬레이션
- **코드 어시스턴트** — 자동완성 (Haiku) / 리팩터링 (Sonnet) / 아키텍처 설계 (Opus)
- **의료 트리아지** — 증상 분류 → 응급 / 일반 / 단순 문의
- **다국어 봇** — 언어 감지 → 언어별 특화 처리
- **이메일 분류기** — 인박스 → 광고 / 업무 / 개인 / 스팸

**함정**: 분류 카테고리 너무 잘게 쪼개면 분류기 정확도 ↓. 보통 3~7개. 분류기 정확도 평가셋 따로 둬야 함.

## 3a. Parallelization — Sectioning (독립 부분 task 동시 실행)

```
                ┌→ LLM (요약 작성)
[긴 문서] → split┼→ LLM (감정 분석)    → 합치기 → [Output]
                └→ LLM (키워드 추출)
```

**언제**: 부분 task 가 서로 독립이고 합칠 수 있을 때. latency 1/N 로 단축.

**예시 프로젝트**:
- **PR 코드리뷰 봇** — 변경사항 → 보안/성능/스타일/테스트 동시 검토 → 통합 코멘트
- **긴 보고서 다측면 분석** — 재무/리스크/시장/경쟁 동시 분석 후 통합
- **채용 이력서 평가** — 기술력/경력/문화핏/커뮤니케이션 동시 채점
- **마케팅 카피 검토** — 톤/문법/SEO/규제준수 동시 점검
- **다국어 동시 번역** — 한국어 → 5개 언어 병렬

## 3b. Parallelization — Voting (같은 task 여러 번 → 합의)

```
              ┌→ LLM (판정 1) ┐
[Input] → 같은 ┼→ LLM (판정 2) ─→ 다수결 또는 평균 → [Output]
              └→ LLM (판정 3) ┘
```

**언제**: 안전성·정확도가 중요한 분류·평가. judge 노이즈 줄이기.

**예시 프로젝트**:
- **콘텐츠 모더레이션** — 댓글 차단 판정 3번 → 2/3 동의해야 차단
- **LLM-as-judge 안정화** — 답변 품질 3번 채점 → 평균
- **부정거래 탐지** — 트랜잭션 여러 모델로 동시 채점
- **의료 진단 보조** — 같은 증상 3번 분석 후 일치도 보고

**함정**: 비용 N배. 다양성 위해 temperature 살짝 다르게 하거나 prompt 변형.

## 4. Orchestrator-Worker (중앙 조정자 + 동적 위임)

```
                ┌→ Worker A
[Input] → Orchestrator (LLM) ←→ Worker B   ←─┐
                ↑                              │
                └── 결과 보고 다음 결정 ───────┘
                ↓
            [Output]
```

중앙 LLM 이 task 를 **동적으로 쪼개서** worker 에 위임하고 결과 보고 다음 단계 결정.

**Routing 과의 차이**: routing 은 한 번 정해서 보내고 끝. orchestrator 는 결과 보고 추가 위임 가능 — 반복적·재귀적.

**언제**: task 의 하위 구조가 미리 안 보이고, 결과 보고 다음을 결정해야 할 때.

**예시 프로젝트**:
- **여행 일정 생성기** — orchestrator 가 항공·호텔·식당·박물관 worker 동시 위임 → 동선 worker 추가 호출 → 통합 일정
- **리서치 어시스턴트** — 질문 분해 → 각 부분 검색·요약 worker → 통합 보고서
- **데이터 분석 봇** — 어떤 분석 필요한지 판단 → SQL/차트/요약 worker 분담 → 보고서
- **코드 마이그레이션** — 영향받는 파일 목록 → 파일별 동시 변환 worker → 통합 PR
- **채용 자동화** — JD 보고 필요 정보 결정 → 후보자별 정보수집 worker

**함정**: orchestrator 자율성이 커질수록 agent 와 경계 흐려짐. max iterations 가드 필수.

## 5. Autonomous Agent (자율 루프)

```
                  ┌─────────────────────────────┐
                  ↓                             │
[user task] → LLM ─→ tool 호출 결정 → 도구 실행 ─┤
                  ↓                             │
              "끝났다" 판단 → [Output]          │
                  └────── 안 끝났으면 다시 ─────┘
```

LLM 이 루프 안에서 매 턴마다 **"도구 부를까 / 끝낼까"** 를 스스로 결정. Phase 2 의 tool use 를 `while` 로 감싸면 그게 agent.

**언제**: task 흐름이 사전에 안 정해지고, 환경 피드백 받아 여러 라운드 적응해야 할 때.

**예시 프로젝트**:
- **코드 디버깅 agent** (Cursor / Claude Code 형) — 에러 → 파일 탐색 → 가설 → 수정 → 재테스트 자율 반복
- **이메일 정리 agent** — 받은편지함 → 자동 분류 → 답장 초안 → 일정 등록
- **데이터 탐색 agent** — 자연어 질문 → SQL → 실행 → 결과 보고 추가 query → 답
- **웹 자동화 agent** — "이 사이트에서 X 해줘" → browser tool 로 자율 navigation
- **DevOps incident response** — 알람 → 로그 검색 → 메트릭 → 가설 → 수정 또는 인간 호출

**핵심 가드**:
- 종료 조건 명확히 (max iterations, 비용 한도, "끝났다" 판단 prompt)
- 도구 적게, description 명확히
- trace 로깅 필수 — agent 디버깅 = trace 분석

**함정**: 무한루프, 비용 폭발, 실패 회복 어려움.

## 한눈에 비교

| | 흐름 통제 | 비용 예측 | 디버깅 | 적응성 |
|---|---|---|---|---|
| 1. Chaining | 코드 (정적) | 매우 좋음 | 쉬움 | 낮음 |
| 2. Routing | 코드 (분기) | 좋음 | 쉬움 | 낮음 |
| 3. Parallel | 코드 (병렬) | 좋음 | 보통 | 낮음 |
| 4. Orchestrator | LLM (동적위임) | 보통 | 어려움 | 높음 |
| 5. Agent | LLM (자율루프) | 나쁨 | 매우 어려움 | 매우 높음 |

위에서 아래로 갈수록 **자율성 ↑, 통제력 ↓.** 본인 task 가 가장 위 패턴으로 풀 수 있는지부터 의심.

## "내 프로젝트는 어느 패턴?"

| 만들고 싶은 게 | 패턴 |
|---|---|
| OCR 필드 추출, 회의록 요약, 번역 파이프라인 | **1. Chaining** |
| 고객지원 분류, 모델 비용 최적화, 다국어 처리 | **2. Routing** |
| 코드 PR 리뷰, 다측면 분석, judge 안정화 | **3. Parallel** |
| 여행 일정, 리서치 보고서, 데이터 분석 | **4. Orchestrator** |
| IDE assistant, 디버깅, 이메일 자동화, 웹 자동화 | **5. Agent** |
